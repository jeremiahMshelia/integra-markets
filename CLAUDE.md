# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

> **Project conventions live in [`PROJECT_RULES.md`](./PROJECT_RULES.md).**
> Read it before changing API clients, auth flows, env vars, error handling,
> or build/deploy config. This file covers what `PROJECT_RULES.md` does not:
> the architecture map, repo layout, legacy hazards, and how to work with
> Claude in this repo.

## Architecture

- **Mobile/Web frontend** ‚Äî Expo SDK 52, React Native 0.76, React 18.
  - Routing: Expo Router (file-based) under `app/`.
  - State: local React state + AsyncStorage; Supabase for auth + user data.
  - Web shell: Next.js (`next.config.mjs`) for landing/preview pages;
    Expo web (`react-native-web`) for the app itself.
  - Distribution: EAS Build/Submit for iOS + Android; Vercel/Fly for web.
- **Backend** (`backend/`) ‚Äî FastAPI on Python 3.11.
  - ORM: tortoise-orm + aerich; aiosqlite locally, Supabase Postgres in prod.
  - NLP: NLTK VADER, TextBlob, FinBERT (torch).
  - AI: Groq (chat/summarization), HuggingFace.
  - Data sources: Alpha Vantage, Kalshi prediction markets, RSS/news
    aggregation (feedparser, aiohttp, BeautifulSoup).
  - Push notifications: exponent-server-sdk ‚Üí Expo push tokens.
  - Deploy targets: Google Cloud Run (primary), plus Dockerfiles for
    Fly.io, Railway, Render.

## Repo layout (canonical)

- `app/` ‚Äî Expo Router screens, components, services, hooks, providers.
- `backend/` ‚Äî FastAPI app, NLP/AI services, DB layer, news pipeline.
- `assets/`, `app-store-assets/` ‚Äî image and store assets.
- `__tests__/`, `backend/tests/` ‚Äî JS/TS and Python test suites.
- `scripts/` ‚Äî build, icon generation, automation.
- `expo-plugins/` ‚Äî custom Expo config plugins.
- `supabase/` ‚Äî SQL migrations, edge function config, setup SQL.

## Common commands

- `npm run dev` ‚Äî start Expo dev server.
- `npm run ios` / `npm run android` ‚Äî native run via Expo.
- `npm test` ‚Äî Jest suite. Single test: `npx jest path/to/file.test.tsx`
  or `npx jest -t "test name"`.
- `npm run lint` / `npm run lint:fix` ‚Äî ESLint.
- `npm run build:ios` / `build:android` ‚Äî EAS production build.
- `npm run submit:ios` / `submit:android` ‚Äî EAS Store submit.
- Backend dev: `cd backend && uvicorn main:app --reload --port 8000`.
- Backend tests: `cd backend && pytest`. Single test:
  `pytest path/to/test_file.py::test_name`.

## Legacy hazards ‚Äî read before editing

- **Top-level `.tsx`/`.js` files** (`MainApp.js`, `App.web.js`,
  `App.web.new.js`, `ai-analysis-overlay.tsx`, `app-icon*.tsx`, etc.)
  are legacy duplicates of files now under `app/`. Edit the `app/`
  versions, not these.
- **Three lockfiles tracked** (`package-lock.json`, `pnpm-lock.yaml`,
  `yarn.lock`). npm is the intended manager; the other two are stale.
  Don't regenerate them ‚Äî if you change deps, update `package-lock.json`
  only.
- **Multiple backend entrypoints** ‚Äî `backend/main.py` is canonical.
  `main_production.py`, `main_integrated.py`, `main_simple_nlp.py` are
  alternate deploy-mode entrypoints; only touch them if you understand
  why they exist.
- **`AIAnalysisOverlay` is currently disabled** (commit `85d01aea`) as
  part of the Build 26 stability rollback. Don't re-enable it without
  also addressing the incompatible dependencies that caused the disable.
- **`ios/` is intentionally minimal** ‚Äî only `IntegraMarkets.xcworkspace`
  is tracked; the rest is regenerated by EAS managed prebuild. Don't
  add native iOS source files here; configure via `app.json` and
  `expo-plugins/` instead.
- **`expo-plugins/disable-privacy-aggregation.cjs`** exists to satisfy
  App Store privacy review ‚Äî don't remove without a replacement plan.
- **Multiple deploy configs coexist** (`Dockerfile`, `Dockerfile.web`,
  `Dockerfile.test`, `fly.toml`, `fly.web.toml`, `railway.toml`,
  `vercel.json`). Cloud Run is primary for backend; check before
  modifying any of these.

## Working with Claude

- When unsure which `App.*.js` / `main_*.py` variant is canonical, ask
  before editing.
- For UI changes, run the Expo dev server and exercise the feature in
  iOS simulator before reporting done; lint/tests alone are not enough.
- For backend changes, run `pytest` and start uvicorn locally to
  confirm imports resolve before claiming success.
- Prefer fixing root causes over flags/fallbacks. The repo already has
  parallel implementations of several features ‚Äî don't add another.
- All conventions in `PROJECT_RULES.md` are load-bearing for the build
  process. If a rule there seems wrong, flag it rather than silently
  diverging.

## Code Review Standards

After completing any implementation, run `/simplify` before presenting code.

Flag:
- Functions longer than 30 lines
- Logic duplicated more than twice (extract to utility)
- Any `any` type usage in TypeScript
- Components with more than 3 props that could be grouped into an object
- Missing error handling on async operations
